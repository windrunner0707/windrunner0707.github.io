使用 ChatGPT、阿里云、K3s 与 GitHub Actions 搭建小型网站

技术 · 2025-06-19 · 11 分钟阅读
Kubernetes GitHub 阿里云 CI/CD 云原生
文章目录

    本文记录如何在一台低配置阿里云 ECS 上，用 K3s 运行前后端服务，并通过 GitHub Actions 完成自动构建与部署。ChatGPT 主要用于辅助生成样例代码、Dockerfile 和 Kubernetes 配置；所有生成内容仍需在本地验证后再提交。

    演示地址：https://www.windrunner0707.tech/

    整体架构

    一次代码提交会经过下面这条链路：

    GitHub push
    ↓
GitHub Actions：编译、构建镜像
    ↓
阿里云 ACR：保存容器镜像
    ↓
SSH 登录 ECS，执行 kubectl
    ↓
K3s：Deployment → Service → Traefik Ingress → HTTPS 域名

    本项目使用的主要组件如下：

    • ChatGPT：辅助搭建 Spring Boot、React + Vite 项目并生成配置文件。
    • 阿里云 ECS：2 vCPU、2 GB 内存、3 Mbps 带宽，用于运行 K3s。
    • 阿里云 ACR：保存前后端容器镜像，减少服务器从 Docker Hub 拉取镜像时的不确定性。
    • K3s：轻量级 Kubernetes 发行版，更适合资源有限的单机环境。
    • GitHub Actions：在推送代码后自动编译、构建镜像并部署。

    本文适合学习和个人项目。生产环境还需要补充高可用、监控、备份、权限隔离与回滚策略。

    一、准备云端资源

    1. 创建 ECS 与 ACR

    我的 ECS 配置如下：

    在阿里云容器镜像服务中创建个人版实例、命名空间和镜像仓库。后续 GitHub Actions 会将构建好的镜像推送到这里。

    同时在 ECS 安全组中按需放行 SSH、HTTP 和 HTTPS 端口。数据库端口不应直接暴露到公网。

    2. 安装 K3s

    服务器访问 GitHub 不稳定时，可以使用 K3s 的国内镜像安装脚本：

    wget https://rancher-mirror.oss-cn-beijing.aliyuncs.com/k3s/k3s-install.sh
chmod +x k3s-install.sh

INSTALL_K3S_MIRROR=cn \
INSTALL_K3S_EXEC="--system-default-registry registry.cn-hangzhou.aliyuncs.com" \
./k3s-install.sh

    这里的两个关键配置分别是：

    • INSTALL_K3S_MIRROR=cn：通过国内镜像获取 K3s 二进制文件。
    • --system-default-registry：为 K3s 系统镜像指定国内镜像仓库。

    安装后检查节点状态：

    sudo kubectl get nodes
sudo systemctl status k3s --no-pager

    节点状态为 Ready 后再继续部署。

    二、准备项目

    本文使用两个代码仓库：

    • 前端：transaction-system-frontend，技术栈为 React + Vite。
    • 后端：transaction-system，技术栈为 Spring Boot。

    在接入 CI/CD 前，应先确保两个项目都能在本地完成编译和打包。不要把“能否构建”的问题留到部署阶段排查。

    1. 前端 Dockerfile

    前端使用多阶段构建：Node.js 负责生成静态文件，Nginx 负责提供 Web 服务。

    # 构建阶段
FROM node:20-alpine AS builder
WORKDIR /app

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# 运行阶段
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80

    相比先复制全部文件再执行 npm install，先复制依赖清单并使用 npm ci 更容易利用 Docker 构建缓存，也能保证依赖版本与锁文件一致。

    2. 后端 Dockerfile

    后端先由 Maven 生成 JAR，再制作只包含运行环境的镜像：

    FROM eclipse-temurin:21-jre
WORKDIR /app

COPY target/transaction-system-0.0.1-SNAPSHOT.jar app.jar

EXPOSE 8080
ENTRYPOINT ["java", "-jar", "/app/app.jar"]

    实际项目可以进一步使用固定版本的基础镜像，并配置非 root 用户、JVM 内存上限和健康检查。

    三、配置 GitHub Actions

    在仓库中创建 .github/workflows/deploy.yml：

    下面以后端为例。工作流使用 Git 提交 SHA 作为镜像标签，避免 latest 带来的版本不可追踪问题。

    name: Build and deploy to K3s

on:
  push:
    branches: [main]

concurrency:
  group: production-deploy
  cancel-in-progress: true

env:
  IMAGE_NAME: transaction-system
  ACR_REGISTRY: crpi-kw9b5zmptdsw1y03.cn-wulanchabu.personal.cr.aliyuncs.com
  ACR_NAMESPACE: windrunner0707

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout source
        uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: "21"
          cache: maven

      - name: Build and test
        run: ./mvnw --batch-mode clean verify

      - name: Log in to ACR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.ACR_REGISTRY }}
          username: ${{ secrets.ACR_USERNAME }}
          password: ${{ secrets.ACR_PASSWORD }}

      - name: Build and push image
        uses: docker/build-push-action@v6
        with:
          context: .
          push: true
          tags: ${{ env.ACR_REGISTRY }}/${{ env.ACR_NAMESPACE }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

      - name: Upload Kubernetes manifest
        uses: appleboy/scp-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          source: deployment.yaml
          target: /home/${{ secrets.SERVER_USER }}/deploy

      - name: Deploy to K3s
        uses: appleboy/ssh-action@v1
        env:
          IMAGE: ${{ env.ACR_REGISTRY }}/${{ env.ACR_NAMESPACE }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SERVER_SSH_KEY }}
          envs: IMAGE
          script: |
            export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
            kubectl apply -f /home/${{ secrets.SERVER_USER }}/deploy/deployment.yaml
            kubectl set image deployment/transaction-system transaction-system="$IMAGE"
            kubectl rollout status deployment/transaction-system --timeout=120s

    需要在 GitHub 仓库的 Settings → Secrets and variables → Actions 中配置：

    • ACR_USERNAME、ACR_PASSWORD：ACR 登录凭据。
    • SERVER_HOST、SERVER_USER：ECS 地址和 SSH 用户。
    • SERVER_SSH_KEY：用于部署的 SSH 私钥。

    不要把密码、私钥或证书直接写入仓库。为部署账户授予完成任务所需的最小权限，并为 ACR 凭据和 SSH 密钥设置轮换策略。

    这条流水线依次完成代码检出、Maven 构建与测试、ACR 登录、镜像构建与推送、配置文件上传，以及 K3s 滚动更新。提交代码后，可以在仓库的 Actions 页面查看每一步日志：

    四、验证部署

    登录 ECS 后检查资源和滚动发布状态：

    export KUBECONFIG=/etc/rancher/k3s/k3s.yaml

kubectl get nodes
kubectl get deployments,pods,services,ingress
kubectl rollout status deployment/transaction-system

    如果 Pod 启动失败，可以继续查看事件和容器日志：

    kubectl describe pod <pod-name>
kubectl logs <pod-name> --tail=200

    五、配置域名与 HTTPS

    1. 域名解析

    在阿里云云解析 DNS 中添加记录，将域名指向 ECS 公网 IP。中国内地服务器对外提供网站服务时，还需要按要求完成备案。

    2. 导入 TLS 证书

    可以在阿里云数字证书管理服务中申请证书：

    下载证书后，将公钥和私钥安全地传到服务器。不要将私钥提交到 Git 仓库，也不要通过聊天工具明文传递。

    在 K3s 中创建 TLS Secret：

    kubectl create secret tls windrunner-tls \
  --cert=www.windrunner0707.tech.pem \
  --key=www.windrunner0707.tech.key \
  --namespace=default

    3. 配置 Ingress

    K3s 默认集成 Traefik，可以通过 Ingress 将域名流量转发到前端 Service：

    apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: frontend-ingress
  namespace: default
spec:
  ingressClassName: traefik
  tls:
    - hosts:
        - www.windrunner0707.tech
      secretName: windrunner-tls
  rules:
    - host: www.windrunner0707.tech
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: transaction-system-frontend
                port:
                  number: 80

    应用配置并验证：

    kubectl apply -f ingress.yaml
kubectl get ingress frontend-ingress
curl -I https://www.windrunner0707.tech/

    六、后续优化方向

    至此，一个从代码提交到 HTTPS 访问的最小部署链路已经完成。接下来可以继续补齐：

    • 使用 readinessProbe 和 livenessProbe 改善发布可靠性。
    • 为前后端配置 CPU、内存请求和上限，避免单个 Pod 占满服务器。
    • 使用 cert-manager 自动签发和续期证书。
    • 增加日志、指标监控、数据库备份和告警。
    • 为 GitHub Actions 固定第三方 Action 的提交摘要，并增加环境审批与回滚流程。
    • 完成前后端联调，将敏感配置统一放入 Kubernetes Secret。
