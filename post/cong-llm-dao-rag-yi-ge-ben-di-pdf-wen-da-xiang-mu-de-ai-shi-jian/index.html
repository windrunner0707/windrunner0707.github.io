从 LLM、LiteLLM 到 RAG：一个本地 PDF 问答项目的 AI 实践

技术 · 2026-07-12 · 12 分钟阅读
人工智能 RAG 架构设计

最近我做了一个本地 PDF 问答 Demo：上传一本电子书，系统提取内容并建立知识库；在页面中选中这本书后，就可以针对书里的内容提问。模型、数据库和应用都运行在本机，不依赖云端 AI 服务。完整项目代码已经上传到 GitHub：windrunner0707/ai-tooling。

这个项目不大，却刚好串起了最近经常出现的几个 AI 概念：LLM、Prompt、Embedding、向量数据库、LiteLLM 和 RAG。它们单独看都不难，真正动手时更重要的问题是：每个组件负责什么，数据如何流动，又怎样减少模型一本正经地“编答案”。

本文不追求搭建一个生产级 AI 平台，而是结合项目中的实际代码，记录我从“调用模型”到“让模型基于本地资料回答”的一次简单实践。

一、先把几个 AI 概念说清楚

1. LLM：会根据上下文继续生成文本的模型

LLM 是 Large Language Model，即大语言模型。可以先把它理解为一个能力很强的文本生成器：接收一组消息，根据训练中学到的语言规律，逐个生成后续 token。

项目使用 Ollama 在本地运行 qwen3:8b。其中 8b 表示模型大约有 80 亿参数，但参数量不是效果的唯一标准。模型版本、量化方式、上下文长度、硬件和任务类型都会影响最终体验。

LLM 有通用知识，也擅长总结、翻译和改写，但它并不知道我刚刚上传的 PDF。即使模型“听说过”这本书，它也可能记忆不完整或直接编造细节。因此，单纯把问题发给模型，并不能可靠地完成私有资料问答。

2. Prompt：给模型的任务说明和上下文

Prompt 不只是用户输入的一句话，也包括系统角色、任务约束、参考资料和输出格式。项目把常用任务做成了模板，例如内容总结、翻译、文本润色、提取待办和代码解释：

return [
    {"role": "system", "content": system_content},
    {"role": "user", "content": prompt},
]

这里的 system 消息定义模型怎样工作，user 消息承载本次输入。模板能让同一种任务的提示词保持稳定，但它不是权限控制，也不能保证模型百分之百遵守规则。重要约束仍应由程序和数据访问层落实。

3. Embedding：把文本变成可比较的数字向量

Embedding（向量表示）会把一段文本转换成一组浮点数。语义相近的文本，向量在空间中的距离通常也更近。

例如，“怎样恢复数据库备份”和“数据库灾难恢复步骤”用词不同，关键词匹配可能漏掉，但它们的语义接近，向量检索更容易把它们联系起来。

项目使用 Ollama 中的 bge-m3:latest 生成 1024 维向量，并在代码中检查返回维度：

embedding = embeddings[0]
if len(embedding) != settings.rag_embedding_dimensions:
    raise RAGServiceError("Embedding 维度与数据库配置不一致")

维度必须和数据库字段 VECTOR(1024) 一致。Embedding 模型一旦更换，向量维度和语义空间都可能变化，已有文档通常需要重新生成向量，而不是只改一个模型名称。

4. 向量数据库：保存向量并找出相似内容

向量数据库的核心能力，是按距离寻找与查询最相近的记录。这个 Demo 没有另外引入专用数据库，而是在 PostgreSQL 中安装 pgvector 扩展：

CREATE TABLE document_chunks (
  id BIGSERIAL PRIMARY KEY,
  document_id BIGINT REFERENCES rag_documents(id) ON DELETE CASCADE,
  content TEXT NOT NULL,
  metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
  embedding VECTOR(1024)
);

CREATE INDEX document_chunks_embedding_hnsw_idx
  ON document_chunks
  USING hnsw (embedding vector_cosine_ops);

这样，文档信息、文本、页码等元数据和向量可以放在同一个数据库中。HNSW 索引用更多存储和建索引成本，换取较快的近似最近邻搜索，适合这个小型实践继续向更大数据量扩展。

5. RAG：先检索资料，再让模型回答

RAG 是 Retrieval-Augmented Generation，即检索增强生成。它不是一种新模型，而是一套应用流程：先从外部知识库检索与问题相关的内容，再把这些内容和问题一起交给 LLM。

可以把它压缩成一句话：

RAG = Retrieval（找资料）+ Augmented Prompt（把资料放进提示词）+ Generation（生成回答）

它的价值不是让模型突然“学会”一本书，而是在每次回答前临时给模型一份开卷资料。知识库内容变化时，不需要重新训练 LLM，只需要更新文档和向量。

6. LiteLLM：统一不同模型的调用入口

LiteLLM 不是 LLM，也不负责从文档中检索内容。它更像模型网关：在应用和具体模型服务之间提供统一的 OpenAI 兼容 API，并把应用使用的模型名称映射到真实后端。

项目中的配置很短：

model_list:
  - model_name: qwen3.8
    litellm_params:
      model: ollama/qwen3:8b
      api_base: http://127.0.0.1:11434

应用只认识 qwen3.8，LiteLLM 再把请求转给本地 Ollama 的 qwen3:8b。以后如果增加其他本地模型或云端服务，可以尽量保持上层调用方式不变，并在网关层统一处理路由、超时、重试、日志和成本统计等问题。

不过，多一层网关也意味着多一个需要部署和排查的组件。只有一个固定模型的小脚本可以直接调用 Ollama；当模型来源变多、多个应用需要统一接口时，网关的价值才更明显。

二、项目整体结构：两条模型调用路径

项目由 React 前端、FastAPI 后端、PostgreSQL + pgvector、Ollama 和可选的 LiteLLM 组成：

React 页面
    |
    v
FastAPI 后端
    |-- 普通聊天 --> LiteLLM --> Ollama qwen3:8b
    |
    |-- PDF 入库 --> 文本提取/OCR --> bge-m3 --> PostgreSQL + pgvector
    |
    `-- PDF 问答 --> pgvector 检索 --> Ollama qwen3:8b

这里有一个值得说明的实现边界：普通聊天通过 LiteLLM 调用模型，而当前 RAG 服务为了直接使用 Ollama 的 embedding 和 chat 接口，没有经过 LiteLLM。

这不影响 RAG 的成立，但形成了两套调用方式。作为学习项目，它能直观看清每一层；如果继续工程化，我会考虑让生成模型统一经过 LLMService，同时保留 embedding 的独立接口。这样模型切换、错误处理和监控不会散落在不同服务中。

三、第一步实践：通过 LiteLLM 调用本地 LLM

后端使用 OpenAI Python SDK，但 base_url 指向本地 LiteLLM：

self.client = AsyncOpenAI(
    api_key=settings.litellm_api_key,
    base_url=settings.litellm_base_url,
    timeout=settings.request_timeout_seconds,
)

response = await self.client.chat.completions.create(
    model=model or settings.default_model,
    messages=messages,
    temperature=temperature,
    max_tokens=max_tokens,
)

这种设计把业务代码和 Ollama 的专有接口解耦了。LLMService 只关心 OpenAI 风格的 messages、模型名和采样参数。网关不可用、请求超时或返回错误状态时，再统一转换为应用自己的 LLMServiceError。

项目还实现了流式输出。模型每生成一小段内容，后端就通过 SSE 发送一个 delta 事件，前端不必等完整答案生成后再显示。这不会让模型真正计算得更快，却能明显改善长回答的等待体验。

temperature 可以粗略理解为输出随机性的控制参数。翻译、抽取和 RAG 问答通常更需要稳定，适合较低值；创意写作可以适当提高。但它不是“正确率旋钮”，降低 temperature 也不能阻止模型基于错误上下文生成错误答案。

四、第二步实践：把 PDF 建成可检索的知识库

PDF 入库不是简单地把整个文件塞给模型，而是经过提取、切分、向量化和存储。

1. 提取文字，必要时使用 OCR

普通 PDF 先由 pypdf 读取文字层。如果所有页面都没有提取到文本，系统会用 Poppler 的 pdftoppm 把页面渲染成图片，再用 Tesseract 做 chi_sim+eng OCR。

PDF
  -> pypdf 提取文字
  -> 无文字层时渲染页面图片
  -> Tesseract OCR
  -> 规范化文本

这个 fallback 让扫描版电子书也能入库，但 OCR 更慢，而且识别质量会直接影响后续检索。RAG 无法自动修复源文本中的错字、漏字和页码错位，所以数据质量仍是整个链路的上限。

2. Chunk：为什么要把长文档切成小块

项目按页处理文本，每个 chunk 最多 1200 个字符，相邻块保留 180 个字符的 overlap：

end = min(start + chunk_size, len(text))
content = text[start:end].strip()
start = max(end - overlap, start + 1)

切分是因为 embedding 模型和 LLM 都有上下文限制，同时小块内容通常更容易精确命中。Overlap 可以减少一句话或一段话刚好被切断造成的信息丢失。

但 chunk 不是越小越好。太小会缺少上下文，太大又会混入无关内容并增加 Prompt 长度。当前按字符和页切分实现简单，后续可以尝试按标题、段落或句子做结构化切分，并用真实问题集比较检索效果。

3. 为每个 chunk 生成 embedding

每个文本块都会调用 bge-m3 生成向量，并和原文、文件名、页码、字符范围一起写入数据库。保留原文是因为向量只适合计算相似度，真正交给 LLM 阅读的仍然是文本内容；保留 metadata 则是为了显示引用和定位来源。

当前实现逐个 chunk 等待 embedding 后再写入数据库，逻辑清楚，但大 PDF 会比较慢。后续可以在模型接口支持的前提下批量生成 embedding，或者引入后台任务和进度反馈。

五、第三步实践：从相似检索到带来源回答

用户在前端选择一本书并提问时，页面会同时提交 question 和 document_id：

{
  "question": "这本书主要讲了什么？",
  "document_id": 2
}

后端先把问题也转换成向量，再使用 pgvector 的余弦距离运算符 <=> 取最接近的 5 个文本块：

SELECT content, metadata, embedding <=> :question_vector AS distance
FROM document_chunks
WHERE document_id = :document_id
ORDER BY embedding <=> :question_vector
LIMIT 5;

document_id 过滤非常重要。它确保选择《A》时不会从《B》里找答案。这类范围限制应在 SQL 查询中实现，而不是只在 Prompt 里告诉模型“不要引用其他书”。前者是数据边界，后者只是生成建议。

检索完成后，系统给每个片段编号，把文件名、页码和正文拼进 Prompt：

你是一个本地 RAG 阅读助手。只根据下面的资料回答问题。
如果资料不足，请直接说资料中没有找到答案。
回答要使用中文，并在关键结论后标注来源编号。

资料:
[1] source=book.pdf, page=12
...

问题: ...

模型温度设置为 0.2，倾向于给出较稳定的回答。接口除了答案，还会返回命中的 chunk ID、页码、距离和文本预览，前端可以把这些来源展示给用户。

这比只返回一段答案更可靠，但“带引用”不等于“引用一定支持结论”。生产场景还需要检查距离阈值、引用和结论是否一致，并用一组已知答案的问题持续评估检索召回率与回答准确性。

六、这次实践让我理解的几个边界

1. RAG 不是微调

RAG 在请求时把外部资料放进上下文，适合经常更新、需要引用的知识。微调则通过训练改变模型参数，更适合调整行为、风格或特定任务能力。把一本新 PDF 加入知识库，通常优先考虑 RAG，而不是重新训练模型。

2. 相似不等于相关，更不等于正确

向量距离只能说明两段文本在 embedding 空间中接近。一个问题可能需要跨页推理，也可能包含人名、版本号等更适合关键词检索的信息。更完整的方案可以加入关键词与向量混合检索、reranker、相似度阈值和查询改写。

3. 本地运行不等于天然安全

模型不出本机减少了数据发送给外部服务的风险，但上传文件、数据库权限、日志、临时 OCR 图片和接口访问仍需要保护。如果应用开放给其他人使用，还要增加文件大小与类型限制、身份认证、资源隔离和恶意 Prompt 防护。

4. RAG 也会产生幻觉

检索只负责提供材料，最终回答仍由生成模型完成。材料不相关、缺失或互相矛盾时，模型依然可能推断过度。项目中的“资料不足就明确说明”和来源展示只能降低风险，不能彻底消除幻觉。

5. Demo 与生产系统之间还有一段路

这个项目已经跑通了完整闭环，但如果用于真实业务，还需要补充异步入库、失败重试、重复文件检测、权限隔离、数据删除、模型与 Prompt 版本、调用指标、评估集以及备份恢复等能力。

七、如何在本地跑起来

可以先克隆项目源码：

git clone https://github.com/windrunner0707/ai-tooling.git
cd ai-tooling

仓库地址：github.com/windrunner0707/ai-tooling。

先启动 PostgreSQL + pgvector，并准备 Ollama 模型：

cd /Users/qiqibai/ai-tooling/rag-postgres
docker compose up -d

ollama pull qwen3:8b
ollama pull bge-m3

然后分别启动后端和前端：

cd /Users/qiqibai/ai-tooling/backend
./scripts/start_backend.sh

cd /Users/qiqibai/ai-tooling/frontend
npm run dev

如果要体验普通聊天的统一模型入口，再启动 LiteLLM：

cd /Users/qiqibai/ai-tooling/litellm-gateway
./scripts/start_litellm.sh

最后访问 http://127.0.0.1:5173，上传 PDF，等待提取和 embedding 完成，选择电子书后即可提问。扫描版 PDF 还需要本机安装 Poppler、Tesseract 和中文语言包。

结语

完成这个 Demo 后，我对这些概念的关系有了更具体的认识：LLM 负责理解和生成语言，Prompt 描述任务，Embedding 把语义变成可检索的向量，pgvector 保存并搜索向量，RAG 把找到的资料交给 LLM，LiteLLM 则为不同模型提供相对统一的调用入口。

真正值得练习的不是记住这些缩写，而是画出一次请求经过了哪些组件，知道每一层能解决什么、不能保证什么。先跑通一个本地、可观察的小闭环，再逐步加入评估、监控和安全控制，比一开始堆叠大量 AI 框架更容易建立可靠的工程判断。
