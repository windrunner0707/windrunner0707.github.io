Kubernetes基础组件详解

技术 · 2025-09-30 · 2 分钟阅读
Kubernetes 云原生

概述

Kubernetes（常简称 K8s）把集群分为控制平面和工作节点。控制平面保存并协调集群的期望状态，工作节点负责运行 Pod。理解这条主线后，各组件的职责会清楚很多。

介绍

控制平面（Control Plane）

控制平面接收声明、保存集群状态，并持续让实际状态向期望状态靠拢，主要包括：

API Server

API Server 暴露 Kubernetes API。kubectl、控制器和其他集群组件都通过它读取或修改资源；它还负责认证、鉴权、准入控制和请求校验。

etcd

etcd 是一致性的分布式键值数据库，用于保存 Kubernetes API 对象。它是控制平面的关键数据存储，因此备份、访问控制和磁盘延迟都很重要。

Scheduler

Scheduler 负责把新建的 Pod 分配到合适的节点上。它会综合考虑节点的 CPU、内存、亲和性、反亲和性等条件，尽量做到资源合理分配。

Controller Manager

Controller Manager 运行多种控制器，持续比较期望状态与实际状态。例如 Deployment 需要三个副本而当前只有两个时，相关控制器会创建新的 Pod 对象，随后再由调度器和 kubelet 接手。

工作节点（Worker Node）

工作节点是真正运行应用的地方，每个节点上都包含以下组件：

kubelet

kubelet 监视分配到本节点的 Pod，并通过容器运行时启动容器、执行健康检查和上报状态。副本数量由控制器维护，不是 kubelet 的职责。

kube-proxy

kube-proxy 在节点上实现 Service 的转发规则，通常使用 iptables 或 IPVS 将虚拟 IP 的流量转到后端 Pod。Pod 网络本身通常由 CNI 插件提供。

Container Runtime

容器运行时负责镜像拉取和容器生命周期管理，并通过 CRI 与 kubelet 交互，常见实现有 containerd 和 CRI-O。Kubernetes 自 1.24 起移除了内置的 dockershim，不能再把 Docker Engine 直接当作 CRI 运行时使用。

Pods

Pod 是 Kubernetes 中最小的调度单元，每个 Pod 包含一个或多个紧密相关的容器。Pod 会共享网络和存储卷，方便容器之间的通信。

把一次部署串起来

提交 Deployment 后，API Server 校验并保存对象；控制器创建所需的 Pod；Scheduler 为未调度的 Pod 选择节点；目标节点上的 kubelet 再调用容器运行时启动容器。Service 流量则由 kube-proxy 维护的转发规则送到可用 Pod。沿着这条链路排查问题，通常比孤立地背组件名称更实用。
