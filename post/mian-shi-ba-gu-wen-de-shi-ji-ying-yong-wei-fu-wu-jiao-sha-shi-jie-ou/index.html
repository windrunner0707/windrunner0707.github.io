面试八股文的实际应用：用绞杀者模式完成商品与库存微服务解耦

技术 · 2025-12-25 · 10 分钟阅读
面试八股文 微服务 架构设计

很多面试题看起来像是在背概念：微服务为什么要独立数据库？ThreadLocal 有什么风险？AOP 适合解决什么问题？配置中心如何实现动态开关？同步调用和异步调用有什么区别？

但真正做过系统改造后会发现，这些“八股文”并不是互不相关的知识点。它们往往会在同一个项目里同时出现，并共同解决一个具体问题。

本文以一次商品与库存微服务解耦为背景，复盘如何使用绞杀者模式、动态 Toggle、AOP、Apollo、ThreadLocal 和 HTTP 接口，在不中断线上业务的前提下，将共享数据库逐步改造成边界清晰的微服务架构。

一、项目背景：服务拆开了，数据库却没有拆开

系统中已经存在商品服务和库存服务。从部署形态上看，它们是两个微服务；但从数据层看，两个服务仍然访问同一个数据库，而且数据库中混杂了商品、库存等多个业务领域的数据表。

这种架构属于“应用层拆分、数据层未拆分”。它在微服务改造初期很常见，但会留下几个问题：

  • 商品服务可以绕过库存服务，直接查询甚至修改库存表；
  • 库存服务也可能依赖商品领域的表结构；
  • 一方修改表结构，另一方可能被动受到影响；
  • 两个服务无法真正独立发布、扩容和演进；
  • 数据库成为隐藏的集成层，服务边界只停留在代码和部署层面。

“微服务独享数据库”这道面试题，在这里不再是一句原则。它真正想解决的是数据所有权：一个领域的数据只能由拥有该领域的服务管理，其他服务必须通过稳定的契约访问它。

因此，本次改造的目标不是简单地把表复制到另一个库，而是让商品服务不再直接读取库存领域的数据。最终，库存数据由库存服务负责，商品服务只能通过 HTTP 接口获取。

二、为什么不能一次性切换

最直接的方案是修改商品服务的代码，把原有数据库查询全部替换成库存服务 HTTP 调用，然后一次发布上线。

但真实生产环境很少允许这样做。新接口可能存在字段映射错误、超时、容量不足或边界场景遗漏。一旦全量切换出现问题，影响的是所有流量；如果回滚还需要重新发版，故障恢复时间会被拉长。

这正是绞杀者模式（Strangler Fig Pattern）的典型使用场景：保留旧能力，在外围逐步建立新能力，让流量分批迁移，确认稳定后再删除旧实现。

在本项目中，两条链路并存：

Toggle 关闭：业务接口 -> 原有 DAO -> 共享数据库
Toggle 开启：业务接口 -> HTTP Client -> 库存服务 -> 库存数据库

旧链路提供稳定基线和快速回退能力，新链路承载未来架构。Toggle 决定当前请求走哪条链路。

这也是灰度发布相关八股文的实际含义：灰度不只是“先发一台机器”，还可以是在同一版本中保留两套实现，通过配置动态选择执行路径。

三、Toggle 设计：把切换能力放在请求上下文中

项目使用 Apollo 管理动态配置。运维或研发人员可以在不重新发版的情况下修改开关，使指定接口在新旧链路之间切换。

核心实现是一个枚举单例：

public enum ToggleUtils {
    INSTANCE;

    private final ThreadLocal<Boolean> toggle = new ThreadLocal<>();

    public void set(Boolean toggle) {
        this.toggle.set(toggle);
    }

    public boolean isOn() {
        return Boolean.TRUE.equals(toggle.get());
    }

    public void remove() {
        toggle.remove();
    }
}

这里同时对应三个常见知识点。

1. 为什么使用枚举单例

Toggle 工具类本身不需要保存多份实例。枚举单例写法简洁，并且由 JVM 保证实例唯一，天然避免反射和反序列化创建额外实例的问题。

不过需要注意：单例只保证 ToggleUtils 实例唯一，并不代表所有请求共享同一个 Toggle 值。真正实现请求隔离的是内部的 ThreadLocal。

2. 为什么使用 ThreadLocal

一次请求会经过 Controller、Service、DAO 或 HTTP Client 等多层调用。如果在每一层方法参数中都增加 boolean toggle，会污染大量接口，而且很多中间方法只负责透传这个参数。

ThreadLocal 把 Toggle 值绑定到当前线程。入口处设置一次，同一线程中的下游代码便可随时读取，不需要逐层修改方法签名。

它适合保存这类与单次请求相关、需要跨层传递的上下文信息，例如 traceId、租户 ID、用户信息和灰度标记。

3. 为什么必须 remove

Web 容器通常使用线程池处理请求。请求结束后，工作线程不会销毁，而是回到线程池等待下一次复用。如果只调用 set() 而不调用 remove()，下一个请求可能读到上一个请求遗留的开关值，产生流量串线。

对于 ThreadLocal，内存泄漏只是风险之一；在这个场景中，更危险的是业务上下文污染。因此必须在 finally 语义下清理。

四、AOP：统一完成开关的读取与清理

如果每个接口都手工读取 Apollo 配置、写入 ThreadLocal，再在结束时清理，代码会重复且容易漏掉。项目使用注解和 AOP 统一处理这类横切逻辑。

整体过程如下：

请求进入
  -> AOP 根据注解取得当前接口的 URI
  -> 读取 Apollo 中该 URI 对应的 Toggle
  -> 写入 ThreadLocal
  -> 执行业务代码
  -> 根据 Toggle 选择 HTTP 或原数据库查询
  -> 请求结束后清理 ThreadLocal

示意代码如下：

@Around("@annotation(toggleRoute)")
public Object route(ProceedingJoinPoint point, ToggleRoute toggleRoute) throws Throwable {
    boolean enabled = toggleService.isEnabled(toggleRoute.uri());
    ToggleUtils.INSTANCE.set(enabled);
    try {
        return point.proceed();
    } finally {
        ToggleUtils.INSTANCE.remove();
    }
}

原设计使用 @Before 获取注解中的 URI，并在 @After 清理。实际编码时更推荐 @Around + try/finally，因为它能更直观地保证：无论业务正常返回还是抛出异常，清理逻辑都会执行。

AOP 在这里没有承载业务规则。它只负责所有接口都需要的通用动作：读取配置、建立上下文和释放上下文。这就是 AOP 适合解决的问题——关注点横跨多个业务模块，但又不属于任何一个具体领域。

五、同步与异步：ThreadLocal 为什么会失效

同步调用中，请求的入口和下游逻辑通常在同一个线程执行，因此可以直接从 ThreadLocal 读取 Toggle：

public StockInfo queryStock(String skuId) {
    if (ToggleUtils.INSTANCE.isOn()) {
        return inventoryClient.queryStock(skuId);
    }
    return legacyStockRepository.queryStock(skuId);
}

但异步任务提交到线程池后，会切换到另一个线程。普通 ThreadLocal 的数据不会自动传播到新线程，所以异步线程中读取到的可能是 null，也可能在错误实现下读到线程池残留的旧值。

更稳妥的处理方式是：在进入异步线程之前，从当前线程取出 Toggle，然后显式作为参数向下传递。

boolean toggle = ToggleUtils.INSTANCE.isOn();

executor.execute(() -> {
    if (toggle) {
        inventoryClient.refreshStock();
    } else {
        legacyStockRepository.refreshStock();
    }
});

显式传参虽然朴素，却最容易理解和验证。若系统中有大量上下文需要在线程池间传播，也可以使用任务装饰器统一复制和清理上下文，但仍要避免直接依赖 InheritableThreadLocal：线程池中的线程通常早已创建，它并不能可靠解决线程复用下的上下文传递问题。

六、从本地查询变成 HTTP 调用，代价不只是多了一次网络请求

Toggle 开启后，原来的进程内数据库查询变成跨服务 HTTP 调用。数据库解耦带来了更清晰的领域边界，也引入了分布式系统的新问题。

1. 超时

HTTP Client 必须设置连接超时和读取超时。没有超时的远程调用可能长期占用业务线程，最终拖垮商品服务的线程池。

超时时间不能简单设置得越长越好。它需要小于上游请求的剩余时间预算，并为重试、降级和响应返回预留空间。

2. 重试与幂等

查询接口通常可以在有限次数内重试，但应只对超时、连接失败等瞬时故障重试，不能对所有异常无差别重试。否则库存服务故障时，重试会放大流量，形成重试风暴。

如果未来迁移写接口，还必须考虑幂等键、重复请求和“请求已成功但响应丢失”等问题。读接口的切换只是解耦的第一步。

3. 熔断、隔离与降级

库存服务异常时，商品服务不能无限等待。应通过线程池或信号量隔离、熔断和限流控制故障范围。

是否允许 Toggle 自动回退到旧数据库，需要谨慎决定。自动回退可以提高可用性，但也可能掩盖新旧数据不一致，并让已经计划下线的数据库依赖长期存在。更稳妥的做法是明确降级策略，并记录每一次回退。

4. 接口契约

共享数据库时，调用方依赖的是表结构；改成 HTTP 后，依赖变成 API 契约。库存服务应返回稳定的领域模型，而不是把数据库字段原样暴露出去。

接口需要明确字段语义、空值规则、错误码、版本兼容策略和分页方式。否则只是把“表耦合”替换成“接口耦合”。

七、如何安全地完成灰度切换

有了 Toggle，并不等于迁移就安全了。完整的切换过程还需要可观测、可验证和可回滚。

阶段一：建立新链路，但默认关闭

先完成库存服务接口和商品服务 HTTP Client，上线时保持 Toggle 关闭。此时线上仍走旧数据库，主要验证代码发布没有副作用。

阶段二：影子读与结果比对

在条件允许时，可以仍以旧链路结果为准，同时异步调用新链路，把两边的结果进行比对，但不影响用户响应。

比对时不能只比较整个 JSON 是否完全相同，还要区分：

  • 真正的数据不一致；
  • 字段默认值或排序差异；
  • 新旧链路读取时间不同造成的短暂差异；
  • 接口转换逻辑造成的精度、单位或空值差异。

阶段三：小流量开启

按接口、租户、门店、用户哈希或机器维度逐步放量。Apollo 负责动态配置，Toggle 负责请求级路由。

重点观察新旧链路的请求量、成功率、P95/P99 延迟、超时率、错误码分布、线程池队列、连接池使用率以及结果差异率。

阶段四：全量切换并保留观察期

新链路全量运行后，旧链路暂时保留，但不再承接正常流量。观察期内如果出现严重问题，可以快速关闭 Toggle 回退。

阶段五：删除旧链路

绞杀式重构的终点不是 Toggle 永久存在。确认新链路稳定后，应删除旧 DAO、共享库账号、相关配置和双路判断代码。

只有撤销商品服务对库存表的数据库权限，数据所有权才算真正收回。否则旧路径虽然不再使用，架构耦合依然存在。

八、这个项目把哪些“八股文”串了起来

回看整个项目，会发现多个面试知识点都对应着真实的工程决策：

八股知识点 项目中的实际作用
微服务独立数据库 明确商品与库存的数据所有权，避免跨领域直接操作表
DDD / 限界上下文 判断哪些表、模型和能力属于商品领域，哪些属于库存领域
绞杀者模式 保留旧链路并逐步迁移流量，降低一次性重构风险
配置中心 Apollo 动态管理开关，无需发版即可放量或回滚
AOP 统一读取接口配置、建立和清理请求上下文
单例模式 提供全局唯一的 Toggle 上下文管理入口
ThreadLocal 在同步调用链中传递请求级灰度标记
线程池与异步编程 解释上下文为何丢失，并决定显式传参或统一包装任务
HTTP / RPC 治理 处理超时、重试、熔断、隔离、限流和接口契约
灰度发布 小流量验证新链路，用指标决定是否继续放量
可观测性 用日志、指标和链路追踪证明新链路是否可靠

所以，“八股文”真正有价值的学习方式，不是记住一个标准答案，而是继续追问三个问题：它解决什么工程问题？它有什么适用边界？使用后又会引入什么新问题？

九、容易踩的坑

最后总结几个在类似改造中尤其需要注意的点：

  1. 把 Toggle 当成永久架构。 Toggle 是迁移脚手架，不是最终业务模型。长期保留会让测试组合成倍增加。
  2. 只切代码，不收数据库权限。 调用方仍有权限直连领域表，就无法阻止未来再次产生耦合。
  3. ThreadLocal 未清理。 在线程池环境中会造成请求上下文串线，必须在 finally 中 remove()。
  4. 异步任务直接读取 ThreadLocal。 切换线程后上下文不会可靠继承，应在提交任务前捕获并显式传递。
  5. 无条件自动回退。 回退可能提高可用性，也可能隐藏数据问题；必须监控并限制。
  6. 只看成功率，不看数据正确性。 HTTP 200 不代表库存结果正确，新旧结果比对是迁移期间的重要证据。
  7. 全量后立即删除旧链路。 应先经过稳定观察期，但也要为最终删除设定明确日期和验收条件。

总结

商品与库存解耦表面上是一次数据库拆分，实质上是数据所有权、服务边界和调用关系的重新建立。

绞杀者模式解决迁移节奏问题，Apollo 和 Toggle 提供动态放量与回滚能力，AOP 负责统一建立请求上下文，ThreadLocal 负责同步链路中的上下文传递，显式参数解决异步线程切换，而超时、重试、熔断和可观测性则为新的 HTTP 链路兜底。

这些知识点单独出现时像是面试题，组合起来时就是一套能够落地的重构方案。真正理解八股文，不是能够背出定义，而是能在复杂约束下知道为什么选它、怎样用它，以及什么时候应该把它删掉。
