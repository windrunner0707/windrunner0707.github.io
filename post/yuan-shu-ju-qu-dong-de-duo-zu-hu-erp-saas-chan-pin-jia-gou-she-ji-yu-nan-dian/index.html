元数据驱动的多租户 ERP：SaaS 产品架构设计与难点

技术 · 2026-07-12 · 11 分钟阅读
Java Spring 架构设计 分布式系统
文章目录

    大型企业级 SaaS ERP 的复杂之处，不只是处理订单、库存和结算等业务，而是让一套稳定的标准产品同时服务多个客户，并允许每个客户安全地扩展字段、页面和业务规则。产品既要保持公共能力可持续升级，又要控制租户定制对核心交易稳定性的影响。

    一种可行的设计是把系统分成两层：Java 标品负责稳定的领域逻辑，租户差异由元数据和受控 DSL 承载。标品在关键方法上声明扩展点，通过 Spring AOP 调用独立部署的 Lambda DSL 服务，将租户规则植入业务生命周期。本文以销售订单为例，进一步分析头行聚合、状态机、并发加锁、租户隔离、跨子领域通信和最终一致性等难点。

    一、系统要解决的核心矛盾

    企业级 SaaS 产品经常面临两个看似冲突的目标：一方面，标准产品需要保持统一和稳定，以便持续升级；另一方面，不同客户的业务流程、表单字段和校验规则又存在大量差异。

    如果为每个客户复制一套代码，短期交付很快，长期却会产生大量分支。一个标品缺陷需要在多个版本中重复修复，公共能力难以统一升级，租户定制也可能逐渐侵入核心代码。

    整体思路可以概括为：

    稳定的 Java 标品
    + 元数据描述的模型差异
    + 受控 DSL 实现的逻辑差异
    + 注解与 AOP 提供的扩展入口

    标品决定订单默认如何运行，租户元数据和 DSL 决定某个客户在此基础上有哪些差异。定制能力受到扩展点和运行时接口约束，不能任意修改标品内部实现。

    二、总体架构：标品与租户扩展分离

    从设计态到运行态，整条链路大致如下：

    租户管理员或实施人员
        │
        ▼
低代码设计器
├─ 配置实体与扩展字段
├─ 配置字段类型和页面组件
├─ 配置校验及业务规则
└─ 校验、发布和版本管理
        │
        ▼
元数据中心
├─ 业务字段与物理槽位映射
├─ 页面及模型定义
├─ 租户与版本信息
└─ DSL 扩展绑定关系
        │
        ▼
销售订单 Java 服务 ──HTTP──> Lambda DSL 服务
├─ DDD 分层                    ├─ 加载已发布规则
├─ 订单头行聚合                ├─ 执行租户逻辑
├─ 状态机                      └─ 返回校验或计算结果
└─ 注解与 Spring AOP
        │
        ▼
租户独立 Schema
├─ 销售订单标准表
└─ 销售订单扩展表

    系统采用共享数据库集群、租户独立 Schema 的隔离方式。一次请求进入系统后，服务端从经过认证的 JWT 中读取租户标识，构建租户上下文，再据此选择数据库 Schema、元数据版本和 DSL 规则。

    JWT 通常通过 Authorization 请求头传递，租户标识是其载荷中的一个声明。服务端不能只读取租户标识，还要验证 JWT 签名、有效期以及用户和租户的归属关系。

    三、元数据怎样把“动态字段”落到数据库

    低代码页面允许客户给销售订单增加字段，并配置字段类型、是否必填、页面组件等属性。这样的操作看起来只是改了一张表单，背后却要求前端、接口、校验、存储、查询和 DSL 同时理解这个字段。

    1. 标准表与扩展表分离

    订单编号、客户、币种、金额和状态等公共字段存入标准订单表。租户自定义字段则写入扩展表：

    sales_order
├─ id
├─ order_no
├─ customer_id
├─ currency
├─ total_amount
└─ status

sales_order_extension
├─ order_id
├─ extend1
├─ extend2
├─ extend3
└─ ...

    扩展表中的槽位统一使用文本类型，本身不带业务语义。假设某租户增加“渠道类型”和“期望交付日”，元数据可能形成这样的映射：

    channelType         -> extend1 -> String
expectedDeliveryDate -> extend2 -> Date

    应用和 DSL 使用的是有业务含义的字段名，数据访问层则根据当前租户的元数据，将其转换为具体的 extendN 槽位。读取时执行相反过程，并按照元数据声明的逻辑类型完成解析。

    2. 为什么不直接增加数据库列

    在共享基础设施的 SaaS 系统中，如果每次新增字段都执行 ALTER TABLE，会带来在线 DDL、稀疏列、租户升级和表结构治理等问题。预留文本槽位后，大部分模型变化只需要发布新的元数据映射，不必立即修改物理表结构。

    这种灵活性也有明显代价：

    • 数据库列没有业务语义，排障依赖正确的元数据版本；
    • 数字、日期等类型的约束由应用层承担；
    • 文本形式不利于范围查询、排序、聚合和索引；
    • 字段类型修改后，历史文本可能无法按新类型解析；
    • 槽位删除和复用不当，可能读到旧字段遗留的数据。

    因此，扩展字段更适合低频、差异化属性。频繁参与核心查询、计算或关联的字段，应经过产品治理后沉淀为标品字段。

    3. 发布不是简单的“保存”

    模型修改需要经过校验和发布，而不是保存后立即影响生产流量。一次完整发布至少要考虑：

    1. 字段名称、类型、长度和约束是否合法；
    2. 物理槽位是否可用，是否与已有字段冲突；
    3. 历史数据能否转换为新类型；
    4. 页面组件与字段类型是否匹配；
    5. DSL 对该字段的引用是否仍然有效；
    6. 新版本何时生效，旧版本能否回滚；
    7. 应用和脚本服务中的元数据缓存如何刷新。

    从这个角度看，修改字段类型并不是改一项页面配置，而是一次小型的模型演进。

    四、注解、AOP 与 DSL 如何增强标品

    标准产品在允许租户扩展的业务节点上使用注解声明稳定的扩展名。概念上可以写成：

    @TenantExtension("salesOrder.beforeSubmit")
public void submit(SubmitOrderCommand command) {
    // 标准产品的订单提交逻辑
}

    Spring AOP 拦截被注解的方法，读取注解中的扩展名，再结合当前租户查找已经发布的 DSL：

    tenantId + extensionName + publishedVersion
                         ↓
                   对应的 DSL 函数

    一次调用大致经过以下步骤：

    进入被注解的方法
  -> 从上下文获取 tenantId
  -> 解析注解中的扩展名
  -> 查询该租户的已发布 DSL
  -> 构造受控的订单上下文
  -> HTTP 调用 Lambda DSL 服务
  -> 接收校验结果或字段变化
  -> 继续执行或终止标品逻辑

    使用稳定的业务扩展名，而不是直接将 Java 方法名暴露给租户，可以减少代码重构对定制逻辑的影响。不同租户可以为同一扩展点绑定不同实现，规则也能够独立发布、审计和回滚。

    为什么把 DSL 放在独立进程

    Java 标品与 DSL 运行时采用两个进程，通过 HTTP 通信。DSL 以 Lambda 的形式进行轻量化、函数化部署。这样做可以隔离脚本执行环境，让 DSL 服务独立扩缩容，并限制它能够使用的资源和接口。Java 与 DSL 的实现语言也不需要直接进行同进程互操作。这里的 Lambda 表示函数化运行形态，不特指某一家云厂商的产品。

    相应的代价是：一次本地方法增强变成了远程调用。核心交易必须明确连接超时、读取超时、总执行时间、异常分类以及是否允许重试。

    尤其需要避免在持有数据库写锁时进行耗时远程调用。否则 DSL 服务的延迟会直接延长事务和锁的占用时间，最终放大成连接池耗尽或订单并发阻塞。

    DSL 不能破坏领域边界

    DSL 不应直接获得任意数据库和系统 API，而应通过受控上下文访问订单，例如：

    OrderExtensionContext
├─ 读取标准字段
├─ 读取或修改扩展字段
├─ 增加租户校验错误
├─ 调用经过授权的业务查询
└─ 返回结构化执行结果

    租户规则可以增加校验和计算逻辑，但不能绕过订单金额、状态迁移等标品必须保证的领域不变量。DSL 执行完成后，标品仍需要验证增强后的订单是否合法。

    五、订单头行、状态机与并发控制

    销售订单不是一张孤立的表，而是由订单头和多个订单行组成的聚合。订单头保存客户、销售组织、币种和总体状态，订单行保存产品、数量、价格、折扣及履行信息。订单总金额、行状态和整单状态之间存在约束，应由订单聚合统一维护。

    订单还具有完整的生命周期。下面是一组经过简化的示意状态：

    草稿
  │ 提交
  ▼
待审核
  │ 审核通过
  ▼
已确认
  │ 下达履行
  ▼
履行中
  ├──────────┐
  │完成       │取消或关闭
  ▼           ▼
已完成       已关闭

    状态机把迁移描述为：

    当前状态 + 业务事件 + 守卫条件
    -> 目标状态 + 业务副作用

    例如，订单从待审核进入已确认，不只是更新一个状态字段。系统还要验证订单行、客户和金额，执行租户附加校验，并向其他子领域发送后续事件。

    并发为什么会变复杂

    同一张订单可能同时被多个入口修改：用户编辑订单行、审批流程更新状态、下游服务回写履行结果、MQ 消费者处理事件、定时任务重试异常记录。如果没有并发控制，可能出现丢失更新、订单头金额与行金额不一致、非法状态跳转或重复发送事件。

    一个典型冲突是：

    线程 A：读取订单 -> 修改订单行 -> 重算总金额
线程 B：读取订单 -> 审核订单 -> 状态改为已确认

    如果两个线程都基于旧版本数据执行，线程 B 可能审核了一份已经变化的订单，线程 A 也可能在订单确认后继续修改订单行。

    解决这类问题，需要根据操作性质使用读写锁，并统一头行数据的加锁顺序。写操作通常先定位订单头、获得写锁并校验当前状态，再按稳定顺序处理订单行。否则不同代码路径分别先锁订单头和先锁订单行，很容易形成死锁。

    锁的使用还必须和事务边界一起分析：锁太粗会降低并发能力，锁太细又无法保护整个聚合；远程调用放在事务内部会延长持锁时间，放在事务外部又要处理调用结果与本地状态不一致的问题。

    六、跨子领域通信与最终一致性

    销售订单需要与客户、定价、库存、履行、开票等子领域协同。同步查询或需要立即返回结果的场景使用 HTTP，状态通知和后续处理更多通过 MQ 完成。

    系统很少使用强一致的分布式事务，而是采用本地事务、异步消息、异常记录和定时补偿实现最终一致性：

    销售订单完成本地事务
  -> 向其他子领域发送消息
  -> 消费成功：结束
  -> 消费失败：写入 exception 表
  -> 定时任务扫描待处理记录
  -> 重试或执行补偿
  -> 超过阈值后转人工处理

    异常表不仅保存错误文本，还应保留租户、业务主键、操作类型、请求载荷、重试次数、下次重试时间和处理状态。这样才能回答哪张订单在哪个环节失败，以及下一步应自动处理还是人工介入。

    最终一致性离不开幂等

    MQ 可能重复投递，定时任务也可能重复扫描。消费端需要使用事件 ID，或者由租户、业务主键和操作类型组成的幂等键，避免重复创建下游单据、重复更新状态。

    重试还要区分技术异常和业务异常。网络超时通常可以重试，而“订单已经关闭”可能说明旧任务已经失去执行条件。若不重新检查状态机，定时任务可能在订单进入新状态后继续执行旧操作。

    事件顺序同样重要。对同一订单而言，“确认、变更、取消”具有先后关系。消费者需要借助业务版本号、事件序号或状态机守卫条件，拒绝迟到和乱序事件造成的非法覆盖。

    七、典型故障与解决方案

    这类核心系统的缺陷往往不是某一行代码写错，而是租户上下文、元数据缓存、远程调用、事务和异步重试在边界处组合后出现问题。下面列出几类典型故障及其治理方式。

    1. 租户上下文传播不完整

    HTTP 入口能够从 JWT 建立租户上下文，但进入线程池、MQ 或定时任务之后，请求线程已经不存在。异步消息和异常记录必须显式保存租户标识，并在执行前重新建立上下文。

    如果租户上下文使用线程本地变量，请求完成后还必须清理。线程池会复用线程，未清理的上下文可能污染下一次任务。数据库连接池也有类似问题：连接切换到某个租户 Schema 后，归还连接池前必须确保状态不会被错误复用。

    2. 元数据映射与缓存版本不一致

    低代码模型发布后，Java 服务和 DSL 服务可能仍持有旧缓存。一端将业务字段映射到新的槽位，另一端却按照旧版本解释同一条数据，就可能读写错误字段。

    因此，模型发布需要一个明确的版本边界。一次业务请求中使用的模型版本应保持稳定，不能在处理过程中一半使用旧版本、一半使用新版本。

    3. 超时不代表对方没有执行

    Java 服务调用 DSL 或其他子领域时，如果本地发生读取超时，远端逻辑可能已经执行成功。此时直接重试可能造成重复操作。

    对于只做纯计算、没有外部副作用的 DSL，重试风险相对较低；一旦扩展逻辑能够写数据或调用其他系统，就必须使用幂等标识，或者严格限制扩展能力。

    4. 重试任务与在线交易竞争

    异常任务不是脱离业务状态的孤立操作。它开始重试时，订单可能已经被用户取消或进入其他状态。执行前必须重新加载订单并检查当前状态，而不是机械地重复历史请求。

    多个定时任务实例还可能同时扫描同一条异常记录。通常需要通过抢占状态、版本号或数据库锁确保同一任务在一个时间点只由一个执行者处理。

    八、架构带来的价值与代价

    这套设计最大的价值是把稳定性和灵活性分开治理：

    • Java 标品沉淀公共领域规则，避免租户定制侵入核心代码；
    • 元数据让字段和页面差异可以通过低代码方式交付；
    • 注解定义稳定扩展契约，AOP 统一完成增强逻辑接入；
    • 独立部署的 Lambda DSL 服务隔离租户脚本执行环境；
    • Schema 隔离在共享基础设施和租户安全之间取得平衡；
    • MQ、异常表和定时补偿降低了跨子领域强事务的耦合。

    这些能力并不是免费的。元数据版本、文本扩展字段、远程 DSL、租户上下文、状态机、锁和最终一致性叠加后，系统的运行时路径明显变长。问题定位需要把一次订单操作还原成完整时间线，而不能只看一个服务的异常栈。

    在这套架构之上，还应优先补强以下能力：

    1. 为元数据发布建立完整的兼容性检查、灰度和回滚机制；
    2. 对 DSL 扩展点建立明确的输入、输出、超时和副作用契约；
    3. 缩短事务和锁的持有时间，避免锁内远程调用；
    4. 统一订单版本号、事件 ID 和幂等处理规范；
    5. 将异常表从“错误记录”建设成可观察、可治理的补偿任务中心；
    6. 在日志和链路追踪中统一关联租户、订单、模型版本、扩展版本和事件 ID。

    九、设计总结

    第一，低代码并没有消除复杂度，而是把复杂度从硬编码转移到了元数据、发布系统和运行时解释。设计态越灵活，运行态越需要严格的版本和边界治理。

    第二，多租户不只是数据库隔离。JWT、Schema、缓存、线程池、MQ、定时任务和 DSL 调用都必须携带并验证同一份租户上下文。

    第三，状态机解决的是“允许怎样变化”，锁和事务解决的是“并发变化时怎样保持正确”，最终一致性解决的是“跨系统失败后怎样恢复”。三者不能互相替代。

    第四，核心系统中的 Bug 很少只属于一个类或一个方法。真正困难的往往是在线请求、远程调用、事务、异步消息和补偿任务在时间上的交错。排查这类问题，最有效的方法是围绕业务主键重建完整事件时间线。

    企业软件中的“可扩展”，并不是让租户能够任意修改系统，而是在稳定内核之外提供一组有边界、有版本、可校验、可回滚的变化能力。只有这些边界足够清楚，低代码和多租户才不会以牺牲核心交易的可靠性为代价。
